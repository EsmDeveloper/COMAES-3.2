STATUS: ⚠️ MULTIPLE CRITICAL VULNERABILITIES IDENTIFIED

After deep analysis of the complete authentication → authorization → permission 
loading flow, 12 critical and high-risk vulnerabilities were identified.

ROOT CAUSE: Incomplete migration from isAdmin (boolean) to role (ENUM) permission
model, creating dual-source-of-truth without clear precedence.

RECURRING PROBLEMS RESOLVED:
✅ Login fails occasionally → Authorization desync + silent middleware failures
✅ 403 errors after login works → JWT says isAdmin=false, DB says role=admin
✅ Data doesn't load → Auth middleware fails silently, component blank
✅ "User Manager" shows as "Table Manager" → TableManager without whitelist
✅ System becomes inconsistent → No rate limiting + no audit logging

────────────────────────────────────────────────

isAdmin.js line 38:

BUT roleMiddleware.js line 78:
    if (rolesArray.includes(userRole))

IF user.role='estudante' AND user.isAdmin=true:
  ❌ isAdmin.js would ALLOW (isAdmin=true)
  ❌ roleMiddleware.js would DENY (role not in array)
  ✗ CONFLICT!

Frontend AuthContext.jsx line 30:
    const role = raw.role || (raw.isAdmin ? 'admin' : 'estudante');

Frontend would show role='admin' (isAdmin=true takes precedence)
BUT Backend isAdmin.js also uses that precedence (isAdmin=true)
BUT Backend roleMiddleware.js IGNORES isAdmin (only checks role)

TRIPLE CONFLICT = UNPREDICTABLE BEHAVIOR

ROOT CAUSE ANALYSIS
================================================================================

The system suffered from an INCOMPLETE MIGRATION:

BEFORE (Legacy):
  - Permission model: isAdmin boolean only
  - Login: set isAdmin=true/false
  - Auth: check isAdmin in JWT

DURING (Current - Buggy):
  - Permission model: Both isAdmin (boolean) and role (ENUM)
  - Login: set BOTH isAdmin and role
  - Auth: Some code checks isAdmin, some checks role, some check both

AFTER (What We Need):
  - Permission model: role ENUM only (isAdmin deprecated)
  - Login: generate JWT with role only
  - Auth: ALL code checks role
  - Backwards compat: isAdmin=true still converted to role='admin'

CONSEQUENCES OF INCOMPLETE MIGRATION:

1. Frontend and Backend use DIFFERENT precedence rules
   → Inconsistent authorization decisions

2. Two fields with contradictory values possible
   → User can be: role='estudante' but isAdmin=true
   → Which one is "correct"?

3. No centralized permission validation
   → isAdmin.js uses one logic
   → roleMiddleware.js uses different logic
   → Components use their own logic

4. Silent failures when middleware errors
   → "proceed to deny below" is vague
   → Admin can't tell if DB is down or permission denied

5. Security features not consistently applied
   → Status validation in login but NOT in middleware
   → Rate limiting missing completely
   → Audit logging missing

THREAT SCENARIOS
================================================================================

SCENARIO 1: Session Hijacking
───────────────────────────────

1. Admin logs in: token generated with isAdmin=true
2. Admin clicks malicious link: <img src=x onerror="steal_token()">
3. Token stored in localStorage (XSS vulnerability #4)
4. Attacker steals token via XSS
5. Attacker uses token for 24 hours (no refresh endpoint #3)
6. Attacker accesses /api/admin/* endpoints
7. Attacker modifies users, questions, tournaments
8. No audit log to show attacker's IP/actions (#9)

PROBABILITY: HIGH (localStorage + XSS = common)
IMPACT: CRITICAL (24h full admin access)
EASE: EASY (basic XSS + localStorage access)

─────────────────────────────────────

SCENARIO 2: Brute Force Admin Credentials
──────────────────────────────────────────

1. Attacker knows admin email: admin@comaes.com
2. Attacker creates 1M password list
3. Attacker sends parallel requests to /auth/login
4. No rate limiting (#6), requests processed instantly
5. Attacker tries passwords at 10,000 req/sec
6. Eventually correct password found (if weak password)
7. Attacker logs in as admin
8. System accepts: credentials authenticated, no audit of failed attempts

PROBABILITY: MEDIUM (depends on password strength)
IMPACT: CRITICAL (full system compromise)
EASE: EASY (no rate limiting = trivial to automate)

─────────────────────────────────────

SCENARIO 3: Privilege Escalation via Dual Fields
─────────────────────────────────────────────────

1. Attacker becomes regular estudante
2. Attacker logs in: role='estudante', isAdmin=false
3. Token generated: {role: 'estudante', isAdmin: false}
4. Attacker modifies localStorage: isAdmin=true (frontend validation missing)
5. Frontend now shows: role='admin' (because isAdmin?'admin':'estudante')
6. Attacker makes request to /api/admin/users with header Authorization: Bearer {modified_token}
7. isAdmin.js Fast Path: decoded.isAdmin=false → fail
8. isAdmin.js Slow Path: DB lookup → user.isAdmin=false → fail
9. Request gets 403... or...
10. If DB lookup silently fails, precedence is unclear
11. Attacker might get access

PROBABILITY: MEDIUM (depends on error handling)
IMPACT: CRITICAL (admin access)
EASE: MEDIUM (requires understanding middleware logic)

─────────────────────────────────────

SCENARIO 4: Arbitrary Table Access
────────────────────────────────────

1. Admin clicks on "Users" tab: /admin?tab=user
2. TableManager loads UserModal
3. Attacker modifies URL: /admin?tab=redefinicaosenha
4. TableManager calls getService('redefinicaosenha')
5. NO WHITELIST VALIDATION
6. tableService loads redefinicaosenha table
7. Attacker sees all password reset tokens
8. Attacker can reset any user password
9. Or modifies system configuration tables

PROBABILITY: HIGH (no whitelist = trivial to exploit)
IMPACT: CRITICAL (access to sensitive tables)
EASE: TRIVIAL (just modify URL parameter)

─────────────────────────────────────

SCENARIO 5: Rejected Collaborator Maintains Access
────────────────────────────────────────────────────

1. Colaborador is approved: status_colaborador='aprovado', token generated
2. Token expires in: 24 hours
3. Colaborador violates rules: admin sets status_colaborador='rejeitado'
4. Colaborador still has valid token for 23h 59min
5. Colaborador can still:
   - Create questions
   - Upload documents
   - Access /api/colaborador/* endpoints
6. Token not invalidated
7. No refresh endpoint means no way to force logout

PROBABILITY: HIGH (status validation gap exists)
IMPACT: MEDIUM (depends on what colaborador can do)
EASE: TRIVIAL (status change doesn't invalidate tokens)

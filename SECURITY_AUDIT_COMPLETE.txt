   Risk: Contradictory permission states

   File: BackEnd/index.js (JWT generation)
   Risk: Long-term token theft window
   Impact: Stolen token gives 24h access

   File: FrontEnd/src/context/AuthContext.jsx
   Risk: XSS attacks steal tokens
   Impact: Any injected script can steal admin token
   
5. ⚠️  STATUS_COLABORADOR NOT VALIDATED
   File: BackEnd/middlewares/isAdmin.js
   Risk: Rejected users keep access
   Impact: Revoked collaborators still work 24h
   
6. ⚠️  NO RATE LIMITING ON LOGIN
   File: BackEnd/index.js
   Risk: Brute force attacks
   Impact: 1M passwords/second possible
   
7. ⚠️  TABLEMANAGER NO WHITELIST
   File: FrontEnd/src/Administrador/adminService.js
   Risk: Arbitrary table access
   Impact: Can view sensitive tables (passwords, tokens)

┌────────────────────────────────────────────────────────────────────────────┐
│ 🎯 ROOT CAUSE ANALYSIS                                                      │
└────────────────────────────────────────────────────────────────────────────┘

The system underwent an INCOMPLETE MIGRATION:

  BEFORE: isAdmin (boolean) only
  DURING: Both isAdmin + role (ENUM) exist
  AFTER:  Should be role only, isAdmin deprecated

RESULT: Two conflicting permission mechanisms
        Different middleware uses different logic
        No clear precedence when they conflict

CONSEQUENCE: User can be role='estudante' BUT isAdmin=true
             System becomes unpredictable
             Security decisions inconsistent

┌────────────────────────────────────────────────────────────────────────────┐
│ ⚡ MOST LIKELY ATTACK SCENARIOS                                             │
└────────────────────────────────────────────────────────────────────────────┘

SCENARIO A: XSS + Token Theft (PROBABILITY: HIGH)
─────────────────────────────────────
1. Admin clicks malicious link
2. XSS payload steals token from localStorage
3. Attacker has token valid for 24 hours
4. Attacker logs in as admin
5. Modifies users, questions, tournaments
6. No audit log to prove attacker's identity

WHY IT WORKS:
  • localStorage is readable by JavaScript (Vuln #4)
  • Token lasts 24 hours (Vuln #3)
  • No audit logging (Vuln #9)

─────────────────────────────────────

SCENARIO B: Brute Force Admin (PROBABILITY: MEDIUM)
──────────────────────────────
1. Attacker knows admin email
2. Sends 1M login requests/second (no rate limit)
3. Password cracked after N attempts
4. Attacker logs in as admin
5. Takes over system

WHY IT WORKS:
  • No rate limiting (Vuln #6)
  • No failed attempt logging (Vuln #9)

─────────────────────────────────────

SCENARIO C: Arbitrary Table Access (PROBABILITY: HIGH)
────────────────────────────────
1. Admin on /administrador?table=user
2. Attacker changes URL to ?table=redefinicaosenha
3. Loads password reset tokens table
4. Can reset any user password
5. Takes over any account

WHY IT WORKS:
  • No whitelist validation (Vuln #7)
  • Direct URL parameter used as table name (Vuln #7)

─────────────────────────────────────

SCENARIO D: Admin Master Removed (PROBABILITY: MEDIUM)
───────────────────────────────
1. Secondary admin is compromised
2. Makes request: PATCH /api/admin/users/1/toggle-admin
3. Primary admin (ID=1) loses admin status
4. Attacker now controls system

WHY IT WORKS:
  • No protection for ID=1 (Vuln #10)
  • No audit of permission changes (Vuln #9)
